# Command Preference & Package Governance System  
**Technical Specification – Unified Multi‑Package‑Manager Architecture**  

---

# 1. Purpose

This repository provides a unified, cross‑ecosystem system for:

1. **Command Preference Micro‑Packages**  
   Small, composable packages that override developer commands (e.g., `npm`, `pip`, `curl`) with policy‑aware wrappers.

2. **Package Governance**  
   A mechanism to:
   - Prefer a tool (`prefer-*`)
   - Force a tool (`force-*`)
   - Block a tool (`block-*`)
   - Remove + block installation of a tool (`eject-*`)

3. **Cross‑Package‑Manager Installation**  
   The same micro‑packages can be installed via:
   - Nix (flakes, Devbox, home‑manager, NixOS)
   - Alpine (APK)
   - Debian/Ubuntu (APT/DEB)
   - Fedora (RPM)
   - Arch (PKGBUILD)
   - Homebrew (Formula)
   - mise (plugin shims)

4. **Alignment with the Standard Developer UX Flow (Devbox‑First)**  
   The system must integrate cleanly with the ADR-defined workflow:
   - `direnv → devbox → just (*-internal) → cargo`

---

# 2. Behavioral Model

There are four behavioral classes:

## 2.1 `prefer-*` (soft guidance)
- Warns when the disfavored tool is used.
- Delegates to the preferred tool **if installed**.
- Does **not** install the preferred tool.
- Does **not** remove or block the disfavored tool.
- Does **not** depend on `eject-*`.

Use case: multi‑project environments, gradual migration.

---

## 2.2 `eject-*` (enforcement engine)
- Removes the disfavored tool **if possible**.
- Blocks future installation of the disfavored tool.
- Shadows the binary with a wrapper.
- Does **not** install any replacement.
- Provides cross‑ecosystem enforcement:
  - APT: `Conflicts`, `Provides`, `Replaces`
  - RPM: `Conflicts`
  - APK: `provides`, `replaces`
  - Arch: `provides`, `conflicts`
  - Nix: wrapper + profile removal
  - Homebrew/mise: shadowing only (no uninstall hooks)

If removal is impossible (e.g., npm bundled with nodejs), `eject-*` must:

- Shadow the binary.
- Block future installs.
- Print a clear explanation:

```
npm is bundled with nodejs and cannot be removed without removing nodejs.
npm has been shadowed and future installs are blocked.
```

---

## 2.3 `force-*` (strict replacement)
- **Depends on `eject-*`**
- Installs the preferred tool.
- Replaces the disfavored tool with a wrapper that delegates to the preferred tool.
- Guarantees deterministic behavior.

Example: `force-pnpm`:
- Removes npm (via eject-npm)
- Installs pnpm
- Replaces `npm` with a wrapper that runs pnpm

---

## 2.4 `block-*` (strict prohibition)
- **Depends on `eject-*`**
- Does **not** install any replacement.
- Replaces the disfavored tool with a wrapper that exits with error.

Example: `block-pip`:
- Removes pip (via eject-pip)
- Blocks future pip installs
- Replaces `pip` with a wrapper that prints a message and exits 1

---

# 3. Dependency Graph (Final)

```
prefer-*     (no dependency)
force-*  →   eject-*
block-*  →   eject-*
eject-*      (base layer)
```

This ensures:

- No duplication of removal logic
- No duplication of conflict metadata
- Clean layering
- Predictable behavior across all ecosystems

---

# 4. Handling Bundled Tools (npm inside nodejs)

Some tools cannot be removed without removing their parent runtime.

Example: npm is bundled with nodejs.

In these cases:

### `eject-npm` must:
- NOT remove nodejs
- Shadow `/usr/bin/npm`
- Block future npm installs
- Explain the limitation
- Exit non-zero

### `force-pnpm` must:
- Still install pnpm
- Still shadow npm
- Still delegate to pnpm
- Still depend on eject-npm

### `block-npm` must:
- Still depend on eject-npm
- Still shadow npm
- Still block execution

This ensures deterministic behavior even when upstream packaging is opinionated.

# 5. Multi‑Project Environments

When different projects use different package managers:

- Project A: Yarn (`yarn.lock`)
- Project B: pnpm (`pnpm-lock.yaml`)
- System: npm installed

The correct package is:

### ✔️ `prefer-pnpm`

Because:

- It does not remove npm
- It does not block npm
- It does not install pnpm
- It only warns when npm is used
- It allows Yarn to coexist
- It allows gradual migration

Later, when refactoring is complete, switch to:

- `force-pnpm`  
or  
- `eject-npm`

# 7. Wrapper Script Requirements

Each wrapper script must:

1. Be POSIX shell (`#!/usr/bin/env sh`)
2. Accept all arguments (`"$@"`)
3. Print a clear message
4. Delegate or block depending on behavior type
5. Contain **no hardcoded paths** (Nix injects absolute paths)

### Example: prefer-pnpm

```sh
#!/usr/bin/env sh
echo "⚠️ Prefer pnpm over npm. Running pnpm instead..."
exec pnpm "$@"
```

### Example: block-npm

```sh
#!/usr/bin/env sh
echo "❌ npm is blocked by policy. Use pnpm or corepack instead."
exit 1
```

### Example: eject-npm (bundled case)

```sh
#!/usr/bin/env sh
echo "❌ npm has been ejected by policy."
echo "npm is bundled with nodejs and cannot be removed without removing nodejs."
echo "npm has been shadowed and future installs are blocked."
exit 1
```

# 8. Nix / Devbox Implementation

## 8.1 Nix derivation pattern

Each micro‑package is a simple wrapper:

```nix
{ pkgs }:

let
  pnpm = pkgs.pnpm;
in
pkgs.writeShellScriptBin "npm" ''
  ${builtins.readFile ../wrappers/npm.prefer-pnpm.sh}
'' // substitute pnpm path
```

## 8.2 Bundle package

```nix
pkgs.symlinkJoin {
  name = "command-governance";
  paths = [
    self.packages.${system}.prefer-pnpm
    self.packages.${system}.prefer-uv
    self.packages.${system}.prefer-corepack
    self.packages.${system}.prefer-devbox
    self.packages.${system}.block-pip
  ];
}
```

## 8.3 Devbox usage

```bash
devbox add github:you/command-governance#prefer-pnpm
devbox add github:you/command-governance#command-governance
```

# 10. Cross‑Package‑Manager Removal Logic

`eject-*` must:

- Remove the package if possible
- Block future installs
- Shadow the binary
- Explain limitations
- Exit non-zero

This logic is implemented via:

| Manager | Mechanism |
|---------|-----------|
| APT | `preinst` removal + `Conflicts` |
| RPM | `%pre` removal + `Conflicts` |
| APK | `pre-install` removal + `replaces` |
| Arch | `install` script removal + `conflicts` |
| Nix | profile removal + wrapper |
| Homebrew | shadow only |
| mise | shadow only |
